# <InternalPageTitle> Qu'est ce que la conteneurisation ? </InternalPageTitle>

## Définition

La conteneurisation consiste à emballer une application et toutes ses dépendances dans une unité autonome appelée **conteneur**. Ce conteneur peut ensuite s'exécuter de manière identique sur n'importe quelle machine disposant de Docker, qu'il s'agisse de votre ordinateur, d'un serveur de test ou d'un serveur de production.

Vous utilisez déjà Docker pour faire tourner votre base de données PostgreSQL. L'objectif de cette section est de **conteneuriser l'ensemble de l'application** : le frontend, l'API et le reverse proxy, en plus de la base de données déjà présente.

Nous allons pour cela créer une conteneurisation complète, qui crée l'application utilisable pour la démonstration finale du projet CAE, selon l'environnement de démo décrit dans la section Environnements.

## Images et conteneurs

- Une **image Docker** est un modèle figé qui décrit comment construire l'environnement d'exécution d'une application. On la crée à partir d'un fichier **`Dockerfile`**.
- Un **conteneur** est une instance en cours d'exécution d'une image. Plusieurs conteneurs peuvent être lancés à partir de la même image.

## Docker Compose

**Docker Compose** permet de définir et de démarrer plusieurs conteneurs ensemble. Plutôt que de lancer chaque conteneur à la main, un fichier `docker-compose.yml` décrit tous les services de l'application, leurs dépendances et leur configuration réseau. Une seule commande suffit ensuite pour tout démarrer :

```bash
docker compose up --build
```

Vous allez déplacer et étendre le `docker-compose.yml` qui se trouve actuellement dans `/api` (et qui ne contient que la base de données) vers la **racine du projet**, pour y inclure tous les services.

# <InternalPageTitle> Les Dockerfiles de l'application </InternalPageTitle>

## Le Dockerfile du frontend

Le frontend React doit être compilé en fichiers statiques (HTML, CSS, JS) puis servi par un serveur web. Nous utilisons **nginx** pour cela, qui est un serveur web léger parfaitement adapté à cet usage.

La construction se fait en deux étapes :
1. Un conteneur Node.js compile l'application (`npm run build`) et produit le répertoire `dist/`.
2. Un conteneur nginx ne récupère que ce répertoire `dist/` et le sert.

Créez un fichier **`Dockerfile`** à la racine de votre répertoire `/frontend` :

```Dockerfile
# Étape 1 : compilation du frontend
FROM node:20-alpine AS build

WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build

# Étape 2 : serveur nginx pour servir les fichiers statiques
FROM nginx:alpine AS prod

WORKDIR /usr/share/nginx/html

COPY --from=build /app/dist .
COPY nginx.conf /etc/nginx/nginx.conf

EXPOSE 5172

CMD ["nginx", "-g", "daemon off;"]
```

Ce Dockerfile copie d'abord uniquement les fichiers de dépendances (`package*.json`), installe les dépendances, puis copie le reste du code source. Cet ordre permet à Docker d'utiliser son cache : si vos dépendances n'ont pas changé, l'étape `npm ci` ne sera pas réexécutée lors des builds suivants.

Créez également un fichier **`nginx.conf`** à la racine de `/frontend`, qui configure nginx pour écouter sur le port 5172 et servir l'application :

```nginx
events {}

http {
    include /etc/nginx/mime.types;

    server {
        listen 5172;

        location / {
            root /usr/share/nginx/html;
            # Renvoie index.html pour toutes les routes (nécessaire pour le routing React)
            try_files $uri $uri/ /index.html;
        }
    }
}
```

La directive `try_files $uri $uri/ /index.html` est essentielle pour les Single Page Applications : elle renvoie `index.html` pour toute URL qui ne correspond pas à un fichier existant, laissant React Router gérer la navigation côté client.

Créez enfin un fichier **`.dockerignore`** à la racine de `/frontend` pour éviter de copier des fichiers inutiles dans l'image :

```
node_modules
dist
.env*
.git
```

## Le Dockerfile de l'API

L'API Spring Boot doit être compilée en JAR puis exécutée dans un environnement Java. Comme pour le frontend, la construction se fait en deux étapes :

1. Un conteneur Maven compile l'application et produit le fichier `.jar`.
2. Un conteneur Java léger ne récupère que ce `.jar` et l'exécute.

Créez un fichier **`Dockerfile`** à la racine de votre répertoire `/api` :

```Dockerfile
# Étape 1 : compilation avec Maven
FROM maven:3.9.9-amazoncorretto-21 AS build

WORKDIR /app

COPY pom.xml .
COPY src ./src

RUN mvn clean package -DskipTests

# Étape 2 : exécution avec un JRE léger
FROM eclipse-temurin:21-jre-jammy AS prod

WORKDIR /app

COPY --from=build /app/target/*.jar app.jar

EXPOSE 3000

CMD ["java", "-jar", "app.jar"]
```

L'image finale ne contient que le JRE (Java Runtime Environment) et le JAR, sans Maven ni le code source. Cela la rend plus légère et plus sécurisée.

Créez également un **`.dockerignore`** à la racine de `/api` :

```
target
.env*
.git
```

Le répertoire `target/` contient les fichiers compilés par Maven sur votre machine. Il ne faut pas le copier dans le conteneur : Maven le recrée lui-même lors du build Docker.

# <InternalPageTitle> Le rôle du reverse proxy </InternalPageTitle>

## Pourquoi un reverse proxy ?

En développement, le serveur Vite joue le rôle de proxy : il intercepte les requêtes vers `/api/...` et les redirige vers `http://localhost:3000`. En production, le serveur Vite n'existe plus — c'est un simple serveur de fichiers statiques (nginx) qui sert le frontend.

Il faut donc un **reverse proxy** qui joue le rôle du proxy Vite en production : un seul point d'entrée sur le port 80 qui redirige les requêtes vers le bon service selon le chemin de l'URL.

Dans notre cas, nous utilisons un conteneur nginx dédié au reverse proxy, distinct du nginx qui sert le frontend :
- Les requêtes vers **`/api/...`** sont redirigées vers l'API (port 3000).
- Toutes les autres requêtes sont redirigées vers le frontend (port 5172).

Grâce à cette architecture, le code frontend n'a pas besoin de changer entre le développement et la production : il fait toujours ses requêtes vers `/api/...`, que ce soit Vite ou nginx qui joue le proxy.

## Configuration du reverse proxy

Créez un fichier **`nginx.conf`** à la **racine du projet** (pas dans `/frontend`, mais au même niveau que `/api` et `/frontend`) :

```nginx
events {}

http {
    include /etc/nginx/mime.types;

    server {
        listen 80;

        # Redirige les requêtes /api/ vers l'API Spring Boot
        location /api/ {
            proxy_pass http://api:3000/;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }

        # Redirige toutes les autres requêtes vers le frontend
        location / {
            proxy_pass http://frontend:5172;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }
    }
}
```

Les noms `api` et `frontend` dans `proxy_pass` correspondent aux noms des services définis dans le `docker-compose.yml`. Docker Compose crée automatiquement un réseau interne qui permet aux conteneurs de se trouver par leur nom de service.

# <InternalPageTitle> Le Docker Compose </InternalPageTitle>

## Structure du projet

Votre projet doit avoir la structure suivante à ce stade :

```
projet/
├── api/
│   ├── Dockerfile          ← nouveau
│   ├── .dockerignore       ← nouveau
│   ├── docker-compose.yml  ← déjà présent (DB uniquement), à supprimer
│   └── src/...
├── frontend/
│   ├── Dockerfile          ← nouveau
│   ├── .dockerignore       ← nouveau
│   ├── nginx.conf          ← nouveau
│   └── src/...
├── nginx.conf              ← nouveau (reverse proxy)
└── docker-compose.yml      ← nouveau à la racine (remplace celui de /api)
```

Le `docker-compose.yml` qui se trouve actuellement dans `/api` ne contenait que la base de données. Nous allons le **remplacer** par un nouveau `docker-compose.yml` à la racine du projet, qui orchestre tous les services.

## Le `docker-compose.yml` complet

Créez le fichier **`docker-compose.yml`** à la racine du projet :

```yaml
services:

  frontend:
    container_name: frontend-demo
    image: frontend-demo
    build:
      context: ./frontend
    ports:
      - "5172:5172"
    depends_on:
      - api

  api:
    container_name: api-demo
    image: api-demo
    build:
      context: ./api
    ports:
      - "3000:3000"
    depends_on:
      - db
    environment:
      SPRING_PROFILES_ACTIVE: demo
      DB_HOST: db
      DB_PORT: 5432
      DB_NAME: ${DB_NAME}
      DB_USER: ${DB_USER}
      DB_PASSWORD: ${DB_PASSWORD}
      JWT_SECRET: ${JWT_SECRET}

  db:
    image: postgres:latest
    container_name: db-demo
    environment:
      POSTGRES_DB: ${DB_NAME}
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    ports:
      - "5433:5432"

  reverse-proxy:
    container_name: reverse-proxy-demo
    image: nginx:alpine
    ports:
      - "80:80"
    depends_on:
      - frontend
      - api
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
```

## Explications des éléments clés

**`depends_on`** définit l'ordre de démarrage des conteneurs. Le service `api` attend que `db` soit démarré avant de se lancer, et `frontend` attend `api`. Attention : cela ne garantit pas que le service est *prêt* (la base de données peut mettre quelques secondes à accepter des connexions après le démarrage du conteneur). C'est une limitation connue qu'on accepte pour le projet CAE.

**`environment`** sur le service `api` fournit les variables d'environnement directement au conteneur, de la même manière que le fichier `.env` le fait en développement. Notez que `DB_HOST` est maintenant `db` (le nom du service Docker), et `DB_PORT` est `5432` (le port interne du conteneur Postgres). La variable `JWT_SECRET` est récupérée depuis le fichier `.env` à la racine du projet grâce à la syntaxe `${JWT_SECRET}`. 
La variable d'environnement `SPRING_PROFILES_ACTIVE` est définie sur `demo` pour que l'API démarre avec le profil de démonstration.

**`volumes`** sur le service `reverse-proxy` monte le fichier `nginx.conf` du projet dans le conteneur. Cela permet de modifier la configuration du reverse proxy sans reconstruire l'image.

## Variables d'environnement à la racine

Créez un fichier **`.env`** à la racine du projet (non commité) pour les secrets nécessaires au `docker-compose.yml` :

```properties
DB_NAME=cae_db
DB_USER=cae_user
DB_PASSWORD=cae_password
JWT_SECRET=votre_secret_jwt
```

Docker Compose charge automatiquement le fichier `.env` situé dans le même répertoire que le `docker-compose.yml`.

## Lancer l'application

Pour construire les images et démarrer tous les services :

```bash
docker compose up --build
```

Pour arrêter et supprimer les conteneurs :

```bash
docker compose down
```

Une fois démarrée, l'application est accessible sur **`http://localhost`** (port 80, servi par le reverse proxy). Vous pouvez tester l'API via **`http://localhost/api/...`**.

# <InternalPageTitle> Construction des images dans la pipeline </InternalPageTitle>

## Intégrer Docker dans la pipeline GitLab

Maintenant que l'application est dockerisée, nous pouvons ajouter un stage **`build`** dans la pipeline GitLab pour vérifier que les images Docker se construisent correctement à chaque push. C'est une validation importante : si un Dockerfile est cassé, la pipeline échoue immédiatement et les développeurs en sont alertés.

Mettez à jour le fichier **`.gitlab-ci.yml`** à la racine du projet :

```yaml
stages:
  - test
  - build
  - images

api test:
  ...

frontend test:
  ...

api build:
  ...

frontend build:
  ...


images:
  stage: images
  image: docker:latest
  services:
    - docker:dind
  script:
    - docker compose build
```

Le stage `images` utilise l'image `docker:latest` et le service `docker:dind` (Docker-in-Docker), qui permettent d'exécuter des commandes Docker dans l'environnement GitLab CI. La commande `docker compose build` construit toutes les images définies dans le `docker-compose.yml`.

## Ce que vous avez maintenant

À la fin de cette section, votre application est entièrement conteneurisée :

- Une **image frontend** : le site statique React servi par nginx sur le port 5172.
- Une **image API** : l'application Spring Boot compilée en JAR et exécutée avec le profil `demo`.
- Un conteneur **base de données** PostgreSQL.
- Un conteneur **reverse proxy** nginx qui sert de point d'entrée unique sur le port 80.

Tout cela démarre en une seule commande : `docker compose up --build`. La pipeline vérifie que les images peuvent être construites à chaque push sur le dépôt.
