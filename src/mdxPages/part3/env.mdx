# <InternalPageTitle> Qu'est ce qu'un environnement ? </InternalPageTitle>

## Introduction

Qu'est-ce qu'on entend par environnement dans le cadre de la conception d'applications d'entreprise ?  
Un environnement est un cadre d'exécution spécifique dans lequel une application tourne, avec des configurations, des ressources et des comportements adaptés à ce contexte.

## Les différents types d'environnements

Il existe plusieurs types d'environnements, parmi lesquels on peut citer :
- **L'environnement de développement** : C'est l'environnement dans lequel les développeurs travaillent au quotidien. L'application tourne sur leur propre machine, avec des outils de debug, des logs verbeux et des données de test.
- **L'environnement de tests** : C'est l'environnement dans lequel les tests automatisés sont exécutés. Il doit être reproductible et stable.
- **L'environnement de staging** : C'est une copie de la production utilisée pour valider les nouvelles fonctionnalités avant leur mise en ligne.
- **L'environnement de production** : C'est l'environnement réel, celui que les utilisateurs finaux utilisent.

Pour le moment, nous avons principalement travaillé en environnement de développement. Nous allons maintenant voir créer un environnement de tests et un environnement spécifique qui servira à la démonstration finale du projet CAE face au client, avec des configurations adaptées à ces contextes.
Nous n'allons pas créer d'environnement de staging ni de production dans le cadre de ce projet, mais sachez que c'est une étape très courante dans les projets d'entreprise.

# <InternalPageTitle> Comment gérer ces environnements ? </InternalPageTitle>

## Le problème central

Comment faire en sorte que **le même code** se comporte différemment selon l'environnement dans lequel il s'exécute ? Par exemple :

- En développement, la base de données tourne dans un conteneur Docker sur votre machine, accessible sur `localhost:5433`. En production, on utilise souvent un serveur de base de données dédié, avec des identifiants de connexion différents et qui doivent rester secrets.
- En développement, les logs SQL sont verbeux pour faciliter le debug. En production, ils sont souvent désactivés pour des raisons de performance et de sécurité.
- En développement, l'application recrée automatiquement le schéma et insère des données de test à chaque démarrage. En production, il est important de ne jamais perdre les données existantes entre plusieurs redémarrages de l'application.

La réponse est d'**externaliser la configuration** : ne jamais coder en dur dans le code source ce qui dépend de l'environnement, mais le placer dans des fichiers de configuration ou des variables d'environnement.

## Deux mécanismes complémentaires

Nous utilisons deux mécanismes qui se complètent :

- **Les variables d'environnement** (via un fichier `.env`) pour les **secrets et paramètres qui varient selon la machine** : identifiants de base de données, clés JWT, URLs. Ces informations ne doivent jamais être commitées dans le dépôt Git.
- **Les profils Spring Boot** pour les **comportements qui changent selon l'environnement** : stratégie DDL, verbosité des logs, initialisation des données. Ces fichiers de configuration, eux, sont versionnés — ils ne contiennent pas de secrets.

# <InternalPageTitle> Comment gérer les variables d'environnement ? </InternalPageTitle>

## Ce que vous avez déjà en place

Depuis la fiche 3, vous avez déjà mis en place la gestion des secrets via un fichier `.env` et la bibliothèque `dotenv-java`. Vous pouvez également retrouver un document de référence pour la mise en place de ce fichier `.env` sur moodle. Votre projet devrait déjà disposer de :

Un fichier **`.env`** à la racine de `/api` (non commité, dans le `.gitignore`) :

```properties
DB_HOST=localhost
DB_PORT=5433
DB_NAME=cae_db
DB_USER=cae_user
DB_PASSWORD=cae
JWT_SECRET=votre_secret_jwt
```

D'un fichier **`application.properties`** qui référence ces variables :

```properties
spring.datasource.url=jdbc:postgresql://${DB_HOST:localhost}:${DB_PORT:5432}/${DB_NAME}
spring.datasource.username=${DB_USER}
spring.datasource.password=${DB_PASSWORD}
```

La syntaxe `${VARIABLE:valeur_par_defaut}` permet de définir une valeur par défaut si la variable n'est pas définie — utile pour ne pas bloquer si le `.env` est absent ou incomplet.

# <InternalPageTitle> Quelles configurations selon l'environnement ? </InternalPageTitle>

## Les profils Spring Boot

Spring Boot permet de définir des **profils** pour adapter la configuration selon l'environnement. Chaque profil correspond à un fichier `application-{profil}.properties` dans `src/main/resources`. Quand un profil est actif, Spring charge d'abord `application.properties` (la base commune), puis superpose `application-{profil}.properties` (les surcharges spécifiques).

Pour notre projet, nous créons trois profils :
- **`dev`** : pour le développement local.
- **`test`** : pour l'environnement de test.
- **`demo`** : pour l'environnement de démonstration.

## Configuration commune (`application.properties`)

Ce fichier contient tout ce qui est identique dans tous les environnements, sans aucun secret en clair :

```properties
# Exécution du backend sur le port 3000 au lieu du port 8080 par défaut
server.port=3000

spring.datasource.driver-class-name=org.postgresql.Driver
spring.jpa.database-platform=org.hibernate.dialect.PostgreSQLDialect

spring.datasource.url=jdbc:postgresql://${DB_HOST}:${DB_PORT}/${DB_NAME}
spring.datasource.username=${DB_USER}
spring.datasource.password=${DB_PASSWORD}
```

## Profil de développement (`application-dev.properties`)

En développement, on veut voir ce qui se passe dans la base de données et repartir sur un schéma propre à chaque redémarrage :

```properties
# Recrée le schéma à chaque démarrage et le supprime à l'arrêt
# Pratique en dev : on repart toujours sur une base propre
spring.jpa.hibernate.ddl-auto=create-drop
spring.jpa.generate-ddl=true

# Logs SQL pour faciliter le debug
spring.jpa.show-sql=true
spring.jpa.properties.hibernate.format_sql=true
```

Le mode `create-drop` signifie que les tables sont recréées à chaque démarrage de l'application et supprimées à l'arrêt. Toutes les données sont perdues entre deux redémarrages — c'est intentionnel en développement, et c'est pour ça qu'on ajoute un initialiseur de données (voir section suivante).

## Profil de test (`application-test.properties`)

En test, on veut un comportement similaire au développement pour la base de données (repartir sur une base propre), mais sans les logs SQL qui polluent la sortie des tests. On souhaite également que la base de données soient remplie avec des données de test spécifiques, différentes de celles utilisées en développement :

```properties
# Recrée le schéma à chaque démarrage et le supprime à l'arrêt
spring.jpa.hibernate.ddl-auto=create-drop
spring.jpa.generate-ddl=true

# Pas de logs SQL en test pour ne pas polluer la sortie des tests
spring.jpa.show-sql=false
spring.jpa.properties.hibernate.format_sql=false
```

Nous verrons dans la section suivante comment insérer des données de test spécifiques à ce profil, pour que les tests puissent s'appuyer sur un jeu de données connu et stable.

## Profil de démonstration (`application-demo.properties`)

La différence entre les environnements de démonstration finale et de tests ne joue au final que sur les données : en démonstration, on veut un jeu de données réaliste et complet pour impressionner le client, tandis qu'en test, on veut un jeu de données minimaliste et contrôlé pour faciliter l'écriture des tests automatisés.
Pour le fichier properties, on peut donc reprendre la même configuration que pour le profil de test.

```properties
# Recrée le schéma à chaque démarrage et le supprime à l'arrêt
spring.jpa.hibernate.ddl-auto=create-drop
spring.jpa.generate-ddl=true

# Pas de logs SQL
spring.jpa.show-sql=false
spring.jpa.properties.hibernate.format_sql=false
```

## Ce qu'il faudrait en production ?

En production, il est crucial de ne jamais perdre les données existantes entre plusieurs redémarrages de l'application. Par conséquent, on utilise le mode `ddl-auto=none` qui signifie que Hibernate n'intervient pas du tout sur le schéma de la base de données : il ne le crée pas, ne le modifie pas, ne le valide pas.
Cela signifie que la base de données doit être initialisée manuellement avant de démarrer l'application en production ! 

Dans une vraie application d'entreprise, c'est une étape critique qui doit être gérée avec soin. Il faut maintenir des scripts d'initialisation de la base de données (DDL et données de référence) à jour. En cas de changement du schéma, il faut créer des scripts de migration qui modifient le schéma existant sans perdre les données. C'est une étape qui demande beaucoup de rigueur et de tests pour éviter les pertes de données en production.
Nous n'allons pas configurer de profil de production dans le cadre de ce projet, mais sachez que c'est une étape très courante et critique dans les projets d'entreprise.

## Lancer l'application avec un profil spécifique

En ligne de commande, vous pouvez activer un profil avec l'option JVM :

```bash
mvn spring-boot:run -Dspring-boot.run.profiles=dev
```

Dans IntelliJ, vous pouvez ajouter cette option dans les configurations de lancement. Pour ouvrir les configurations de lancement, sélectionnez "Edit Configurations..." dans le menu déroulant à côté du bouton de lancement, puis ajoutez l'option suivante dans "VM options" :

```
-Dspring.profiles.active=dev
```

Cette configuration lancera l'application avec le profil `dev` actif, ce qui signifie que les propriétés de `application-dev.properties` seront utilisées en plus de celles de `application.properties`.

Il est également possible de créer plusieurs configurations de lancement dans IntelliJ, chacune avec un profil différent (`dev`, `test`, `demo`), pour faciliter le basculement entre les environnements.

# <InternalPageTitle> Comment gérer les données selon l'environnement ? </InternalPageTitle>

## Le problème

Avec `ddl-auto=create-drop`, le schéma de la base de données est recréé vide à chaque démarrage. C'est pratique pour repartir proprement, mais cela signifie que vous devez réinsérer vos données de test à la main à chaque fois — ce qui devient vite fastidieux.

La solution est d'écrire un **initialiseur de données** qui s'exécute automatiquement au démarrage et insère les données nécessaires, mais **uniquement pour un certain profil**.

## Initialiseur avec `CommandLineRunner` et `@Profile`

Spring Boot permet d'exécuter du code au démarrage via l'interface `CommandLineRunner`. Combinée à l'annotation `@Profile`, on peut restreindre son exécution à un profil spécifique.

Voici un exemple concret pour une application de pizzeria :

```java
@Component
@Profile("test") // Ce bean n'existe QUE si le profil "test" est actif
public class DataInitializer implements CommandLineRunner {

  private final PizzaRepository pizzaRepository;
  private final UserRepository userRepository;
  private final PasswordEncoder passwordEncoder;

  public DataInitializer(
      PizzaRepository pizzaRepository,
      UserRepository userRepository,
      PasswordEncoder passwordEncoder) {
    this.pizzaRepository = pizzaRepository;
    this.userRepository = userRepository;
    this.passwordEncoder = passwordEncoder;
  }

  @Override
  public void run(String... args) {
    // Données de test
    pizzaRepository.save(new Pizza("Margherita", "Tomate, mozzarella", 10.50));
    pizzaRepository.save(new Pizza("Regina", "Tomate, mozzarella, jambon, champignons", 12.00));
    pizzaRepository.save(new Pizza("Végétarienne", "Légumes de saison", 11.00));

    // Un utilisateur admin pour tester
    User admin = new User();
    admin.setUsername("admin");
    admin.setPassword(passwordEncoder.encode("admin123")); // Toujours hasher !
    admin.setRole("ADMIN");
    userRepository.save(admin);

    System.out.println(">>> Données de tests initialisées.");
  }
}
```

Points importants :
- **`@Profile("test")`** : Spring ignore complètement cette classe si le profil `test` n'est pas actif. En développement, aucune donnée de test ne sera insérée.
- **Les mots de passe doivent toujours être hashés**, même pour les données de test.

De cette façon, à chaque fois que vous démarrez l'application en profil `test`, vous avez un jeu de données connu et stable. Cela vous permet d'écrire des tests automatisés qui s'appuient sur ces données, sans avoir à les insérer manuellement à chaque fois.

## Adapter les données selon le profil

Si vous avez besoin de jeux de données différents selon les profils, créez simplement un initialiseur par profil :

```java
@Component
@Profile("dev")
public class DevDataInitializer implements CommandLineRunner {
  // Quelques données minimales pour le développement quotidien
}
```

```java
@Component
@Profile("test")
public class TestDataInitializer implements CommandLineRunner {
  // Jeu de données pour les tests automatisés
}
```

```java
@Component
@Profile("demo")
public class DemoDataInitializer implements CommandLineRunner {
  // Jeu de données pour la démonstration finale face au client
}
```

## Initialisation via SQL

Parfois, certaines données de référence doivent se trouver dans la base de données dès le départ, dans la plupart des profils. Pour ces données, il est courant d'utiliser plutôt des scripts SQL d'initialisation. 

Placez un fichier `src/main/resources/data.sql` contenant les instructions SQL d'ajout de ces données communes et contrôlez son exécution dans le profil concerné :

```properties
# Dans application-dev.properties — exécuter le script au démarrage
spring.sql.init.mode=always
```

```properties
# Dans application-prod.properties — ne jamais exécuter le script
spring.sql.init.mode=never
```

# <InternalPageTitle> Comment générer les artefacts ? </InternalPageTitle>

## Qu'est-ce qu'un artefact ?

Un **artefact** est le résultat d'un processus de build : c'est le fichier prêt à être distribué ou exécuté. Dans notre projet, deux artefacts sont produits :

- Un fichier **`.jar`** pour le backend Spring Boot — un exécutable Java autonome qui contient l'application et toutes ses dépendances.
- Un répertoire **`dist/`** pour le frontend React — un ensemble de fichiers HTML, CSS et JavaScript optimisés, prêts à être servis par n'importe quel serveur web statique.

## Générer le JAR du backend

La commande suivante, à exécuter dans le répertoire `/api` :

```bash
mvn clean package
```

Elle nettoie les builds précédents, compile le code, exécute les tests, puis assemble le tout dans un fichier `.jar` dans le répertoire `target/`. Le nom exact dépend de ce qui est défini dans votre `pom.xml` — quelque chose comme `api-0.0.1-SNAPSHOT.jar`.

Pour ignorer les tests lors de la génération de l'artefact final (par exemple lors de la construction d'une image Docker) :

```bash
mvn clean package -DskipTests
```

Le JAR peut ensuite être exécuté directement avec un profil :

```bash
java -jar target/api-0.0.1-SNAPSHOT.jar --spring.profiles.active=demo
```

## Générer le site statique du frontend

La commande suivante permet de construire le site statique, à exécuter dans le répertoire `/frontend` :

```bash
npm run build
```

Elle compile le TypeScript, optimise et minifie le JavaScript et le CSS, et place tous les fichiers dans le répertoire `dist/`. Ce répertoire contient tout ce qu'il faut pour servir l'application : un `index.html`, des fichiers JS et CSS, et les assets statiques.

Avant ça, n'oubliez pas d'installer les dépendances si ce n'est pas déjà fait :

```bash
npm ci
```

On utilise `npm ci` plutôt que `npm install` pour garantir une installation propre et reproductible. Cette installation est basée sur le fichier `package-lock.json`, assurant ainsi la cohérence des dépendances et la reproductibilité. Assurez-vous que le fichier `package-lock.json` est à jour et commité dans le dépôt Git pour que cela puisse fonctionner correctement.

## Génération des artefacts dans la pipeline

Dans la pipeline GitLab, la génération des artefacts peut s'intégrer dans un stage `build`. Les fichiers produits sont ensuite disponibles en téléchargement depuis l'interface GitLab.

```yaml
stages:
  - test
  - build

# stages de test tels que définis précédemment

api test:
  ...

frontend test:
  ...

api build:
  stage: build
  image: maven:3.9.9-amazoncorretto-21
  script:
    - cd api
    - mvn clean package -DskipTests
  artifacts:
    paths:
      - api/target/*.jar

frontend build:
  stage: build
  image: node:20-alpine
  script:
    - cd frontend
    - npm ci
    - npm run build
  artifacts:
    paths:
      - frontend/dist/
```

## Résumé

| Composant | Commande | Artefact produit |
|---|---|---|
| **API (Spring Boot)** | `mvn clean package` | `target/*.jar` |
| **Frontend (React/Vite)** | `npm run build` | `dist/` |
