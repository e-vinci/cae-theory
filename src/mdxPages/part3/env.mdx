# <InternalPageTitle> Qu'est ce qu'un environnement ? </InternalPageTitle>

## Introduction

Qu'est-ce qu'on entend par environnement dans le cadre de la conception d'applications d'entreprise ?  
Un environnement est un ensemble de ressources matérielles et logicielles qui permettent à une application de fonctionner.  

## Les différents types d'environnements

Il existe plusieurs types d'environnements, parmi lesquels on peut citer :
- **L'environnement de développement** : C'est l'environnement dans lequel les développeurs travaillent pour écrire le code de l'application. Il est généralement installé sur les ordinateurs des développeurs.
- **L'environnement de tests** : C'est l'environnement dans lequel les tests unitaires et les tests d'intégration sont exécutés. 
- **L'environnement de staging** : C'est l'environnement dans lequel l'application est déployée pour être testée par les utilisateurs finaux avant sa mise en production.
- **L'environnement de production** : C'est l'environnement dans lequel l'application est déployée pour être utilisée par les utilisateurs finaux.

Notons que l'environnement de développement est généralement installé sur les ordinateurs des développeurs, tandis que les environnements de tests, de staging et de production sont généralement installés sur des serveurs.

 # <InternalPageTitle> Comment gérer ces environnements ? </InternalPageTitle>

## Introduction

La gestion des environnements est une tâche importante dans le développement d'applications d'entreprise. Elle consiste à s'assurer que chaque environnement dispose des ressources nécessaires pour fonctionner correctement et à isoler les environnements les uns des autres pour éviter les conflits.

## Différentes approches de gestion des environnements

Il existe plusieurs approches pour gérer les environnements, parmi lesquelles on peut citer :
- **Serveurs dédiés physiques** : Utilisation de serveurs physiques dédiés pour chaque environnement.
- **Serveurs virtualisés** : Utilisation de machines virtuelles (VM) pour créer des environnements isolés sur des serveurs physiques partagés.
- **Conteneurisation** : Utilisation de conteneurs (par exemple, Docker) pour isoler les applications et leurs dépendances dans des environnements légers et portables.

Nous allons voir les avantages et les inconvénients de chaque approche dans les sections suivantes.

## Avantages et inconvénients des différentes approches

### Serveurs dédiés physiques

**Avantages** :
- Performance optimale car les ressources ne sont pas partagées.
- Isolation complète entre les environnements.

**Inconvénients** :
- Coût élevé en termes de matériel et de maintenance.
- Scalabilité limitée par le nombre de serveurs physiques disponibles.
- Complexité de gestion des serveurs, donc cela nécessite des administrateurs systèmes.

### Serveurs virtualisés

**Avantages** :
- Meilleure utilisation des ressources matérielles.
- Facilité de gestion et de déploiement des environnements.
- Scalabilité améliorée par la création de nouvelles machines virtuelles.
- Ne nécessite pas de gérer le matériel physique. Cela peut être fait par un fournisseur de cloud.

**Inconvénients** :
- Performance légèrement inférieure à celle des serveurs dédiés physiques en raison de la surcharge de virtualisation.
- Complexité de gestion des hyperviseurs et des machines virtuelles (VM).  
NB : l'hyperviseur distribue à chaque VM les ressources qui ont été allouées et gère la planification des ressources des VM selon les ressources physiques disponibles.

### Conteneurisation
**Description** : Utilisation de conteneurs (par exemple, Docker) pour isoler les applications et leurs dépendances dans des environnements légers et portables.

**Avantages** :
- Démarrage rapide et utilisation efficace des ressources.
- Portabilité et consistance entre les environnements de développement, de test, de staging et de production.
- Facilité de déploiement et de mise à l'échelle.

**Inconvénients** :
- Isolation moins complète que les VM, car les conteneurs partagent le même noyau du système d'exploitation.
- Nécessite une bonne gestion des orchestrateurs de conteneurs (par exemple, Kubernetes, Docker Swarm...).  
NB : Un orchestrateur de conteneurs est un système qui gère automatiquement les conteneurs, y compris leur déploiement, leur mise à l'échelle, leur mise en réseau et leur gestion de cycle de vie.

## Quelle approche choisir pour la gestion des environnements ?

La gestion des environnements peut se faire de différentes manières, chacune ayant ses avantages et ses inconvénients. Le choix entre serveurs dédiés physiques, serveurs virtualisés et conteneurisation dépend des besoins spécifiques du projet en termes de performance, de coût, de scalabilité et de facilité de gestion.

Néanmoins, lorsque l'on fait du DevOps, la conteneurisation est souvent privilégiée pour sa facilité de déploiement, sa portabilité et sa scalabilité. Elle permet de créer des environnements cohérents et reproductibles, ce qui facilite le développement, les tests et le déploiement des applications.

De plus, la conteneurisation est un choix naturel pour les applications modernes car elle permet :
- de simplifier la gestion des différents composants (frontend, api, base de données...) d'une application d'entreprise, car chaque composant peut être encapsulé dans un conteneur distinct, chacun ayant ses propres dépendances et configurations.
- de réunir différents composants dans un système de multi-conteneurs, ce qui facilite les communications entre les composants et la gestion des dépendances.
- de pouvoir réutiliser un même conteneur pour plusieurs environnements (dev, test, staging, prod), ce qui permet de garantir la cohérence des environnements et de réduire les risques d'erreurs de configuration.
- de réduire les coûts d'infrastructure en optimisant l'utilisation des ressources matérielles et en permettant une gestion plus efficace des environnements.

## Environnements dans le projet CAE

Dans le cadre de ce cours, nous allons utiliser la conteneurisation pour gérer les environnements. Nous souhaitons avoir chaque composant de l'application (frontend, api, base de données, reverse proxy) encapsulé dans un conteneur distinct, avec une gestion multi-conteneurs à l'aide de **Docker Compose**.

Les environnements pertinents pour notre projet sont :
- **L'environnement de développement** : L'API Spring Boot et le frontend sont exécutés directement sur la machine du développeur (sans Docker). Seule la base de données PostgreSQL est dockerisée pour simplifier sa mise en place.
- **L'environnement de tests local** : Tous les composants (frontend, API, base de données, reverse proxy) sont dockerisés et orchestrés via **Docker Compose**. Cet environnement reproduit fidèlement ce que serait un environnement de production.

# <InternalPageTitle> Comment gérer les variables d'environnement ? </InternalPageTitle>

## Introduction

Une application d'entreprise a besoin de se comporter différemment selon l'environnement dans lequel elle s'exécute. Par exemple, l'URL de la base de données en développement ne sera pas la même qu'en production. Il est donc essentiel de gérer correctement les variables d'environnement pour adapter le comportement de l'application selon l'environnement.

Une règle fondamentale est de **ne jamais coder en dur** des informations sensibles ou spécifiques à un environnement (mots de passe, URLs, secrets JWT...) dans le code source. Ces informations doivent être externalisées dans des variables d'environnement ou des fichiers de configuration non versionnés.

## Gestion des variables d'environnement côté backend (Spring Boot)

Spring Boot offre un mécanisme puissant pour gérer les variables d'environnement à travers le fichier `application.properties` (ou `application.yml`) et les **profils**.

### Le fichier `.env` et `spring-dotenv`

Pour les secrets (mots de passe de base de données, clés JWT...), il est recommandé d'utiliser un fichier **`.env`** à la racine du projet. Ce fichier ne doit **jamais** être commité dans le dépôt Git (il doit être ajouté au `.gitignore`).

La bibliothèque **`spring-dotenv`** permet de charger automatiquement ces variables depuis le fichier `.env` vers l'environnement Spring Boot.

Exemple de fichier `.env` :

```properties
DB_HOST=localhost
DB_PORT=5432
DB_NAME=cae_db
DB_USER=cae_user
DB_PASSWORD=cae
JWT_SECRET=verysecretkeyandverylongsoastobeabletosustainabrute-force-attack
```

Dans `application.properties`, on référence ces variables :

```properties
spring.datasource.url=jdbc:postgresql://${DB_HOST:localhost}:${DB_PORT:5432}/${DB_NAME}
spring.datasource.username=${DB_USER}
spring.datasource.password=${DB_PASSWORD}
```

La syntaxe `${VARIABLE:valeur_par_defaut}` permet de définir une valeur par défaut si la variable n'est pas définie.

### Les profils Spring Boot

Spring Boot permet de définir des profils pour adapter la configuration selon l'environnement. Les profils sont définis dans des fichiers de configuration séparés :

- **`application.properties`** : Configuration commune à tous les environnements.
- **`application-dev.properties`** : Configuration spécifique à l'environnement de développement.
- **`application-prod.properties`** : Configuration spécifique à l'environnement de production (ou de tests avec Docker).

Par exemple, le fichier `application-dev.properties` pourrait contenir :

```properties
# En développement, l'API écoute sur le port 3000 de l'hôte
server.port=3000
spring.datasource.url=jdbc:postgresql://localhost:5432/cae_db
```

Et le fichier `application-prod.properties` :

```properties
# En production (dans Docker), la base de données est accessible via le nom du service Docker
spring.datasource.url=jdbc:postgresql://db:5432/cae_db
```

Pour activer un profil Spring Boot, on peut :
- Le spécifier au démarrage de l'application : `java -jar app.jar --spring.profiles.active=prod`
- Le définir dans le `docker-compose.yml` via l'`entrypoint` : `["java", "-jar", "app.jar", "--spring.profiles.active=prod"]`

## Gestion des variables d'environnement côté frontend (Vite / React)

Vite, le bundler utilisé pour notre frontend React, offre un mécanisme similaire pour gérer les variables d'environnement.

### Les fichiers `.env` de Vite

Vite charge automatiquement les variables d'environnement depuis des fichiers `.env` selon l'environnement :

- **`.env`** : Variables chargées dans tous les environnements.
- **`.env.development`** : Variables chargées uniquement en mode développement (lors de `npm run dev`).
- **`.env.production`** : Variables chargées uniquement en mode production (lors de `npm run build`).

**Important** : Pour être exposées dans le code React, les variables doivent commencer par le préfixe **`VITE_`**.

Exemple de `.env.development` :

```properties
VITE_API_BASE_URL=http://localhost:3000
```

Exemple de `.env.production` :

```properties
VITE_API_BASE_URL=/api
```

Dans le code React, on accède à ces variables via `import.meta.env` :

```tsx
const apiBaseUrl = import.meta.env.VITE_API_BASE_URL;
const response = await fetch(`${apiBaseUrl}/pizzas`);
```

### La variable `import.meta.env.DEV`

Vite expose également la variable `import.meta.env.DEV` qui vaut `true` en mode développement et `false` en mode production. Cela permet d'adapter le comportement du frontend selon l'environnement sans avoir à définir une variable explicite :

```tsx
const apiBaseUrl = import.meta.env.DEV ? '/api' : '/api';
```

En développement, le proxy de Vite (configuré dans `vite.config.ts`) redirige les requêtes `/api` vers `http://localhost:3000`. En production (dans le conteneur Docker), c'est le reverse proxy nginx qui s'en charge.

## Variables d'environnement dans Docker Compose

Dans le fichier `docker-compose.yml`, on peut passer des variables d'environnement à chaque service via la clé `environment` :

```yaml
services:
  db:
    image: postgres:latest
    environment:
      POSTGRES_DB: cae_db
      POSTGRES_USER: cae_user
      POSTGRES_PASSWORD: cae
```

On peut également référencer des variables définies sur la machine hôte ou dans un fichier `.env` situé à la racine du projet :

```yaml
services:
  api:
    environment:
      DB_PASSWORD: ${DB_PASSWORD}
      JWT_SECRET: ${JWT_SECRET}
```

Docker Compose charge automatiquement le fichier `.env` situé à la racine du projet. Ce fichier ne doit pas être commité dans le dépôt Git.

# <InternalPageTitle> Quelles configurations selon l'environnement ? </InternalPageTitle>

## Introduction

Les différents environnements (développement, tests, production) nécessitent des configurations différentes pour les composants de l'application. Voici un tableau récapitulatif des principales différences de configuration selon l'environnement :

| Aspect | Développement | Tests (Docker local) |
|---|---|---|
| **Backend** | Lancé directement via IntelliJ / Maven | Lancé dans un conteneur Docker |
| **Frontend** | Serveur Vite (`npm run dev`) | Site statique servi par nginx (dans Docker) |
| **Base de données** | Conteneur Docker, port exposé sur `localhost` | Conteneur Docker, accessible uniquement depuis le réseau Docker interne |
| **Proxy** | Proxy Vite (dans `vite.config.ts`) | Reverse proxy nginx (conteneur Docker) |
| **Profil Spring** | `dev` (ou aucun) | `prod` |
| **URL de la DB** | `jdbc:postgresql://localhost:5432/cae_db` | `jdbc:postgresql://db:5432/cae_db` |

## Configuration du backend selon l'environnement

### Environnement de développement

En développement, l'API Spring Boot s'exécute directement sur la machine du développeur. La base de données PostgreSQL est accessible sur `localhost` car son port est exposé par le conteneur Docker.

Le fichier `application-dev.properties` (ou `application.properties` si aucun profil n'est activé) :

```properties
server.port=3000
spring.datasource.url=jdbc:postgresql://${DB_HOST:localhost}:${DB_PORT:5432}/${DB_NAME:cae_db}
spring.datasource.username=${DB_USER:cae_user}
spring.datasource.password=${DB_PASSWORD:cae}
```

### Environnement de tests avec Docker

Lorsque l'application est entièrement dockerisée, les services communiquent entre eux via le réseau interne de Docker Compose. Le nom du service `db` dans le `docker-compose.yml` est utilisé comme hostname pour la base de données.

Le fichier `application-prod.properties` :

```properties
spring.datasource.url=jdbc:postgresql://db:5432/${DB_NAME:cae_db}
spring.datasource.username=${DB_USER:cae_user}
spring.datasource.password=${DB_PASSWORD:cae}
```

Le profil `prod` est activé via l'`entrypoint` dans le `docker-compose.yml` :

```yaml
entrypoint: ["java", "-jar", "app.jar", "--spring.profiles.active=prod"]
```

## Configuration du frontend selon l'environnement

### Environnement de développement

En développement, le frontend utilise le serveur de développement Vite. Ce serveur intègre un **proxy** qui redirige les requêtes API vers l'API Spring Boot.

Configuration dans `vite.config.ts` :

```ts
server: {
  proxy: {
    '/api': {
      target: 'http://localhost:3000',
      changeOrigin: true,
      rewrite: (path) => path.replace(/^\/api/, ''),
    },
  },
},
```

Ainsi, toute requête du frontend vers `/api/pizzas` est automatiquement redirigée vers `http://localhost:3000/pizzas` par Vite.

### Environnement de tests avec Docker (production)

En production, le frontend est un site web statique (fichiers HTML, CSS, JS) servi par **nginx**. Le serveur de développement Vite n'existe plus. C'est le **reverse proxy nginx** (un conteneur dédié dans le Docker Compose) qui prend en charge la redirection des requêtes API vers le backend.

Puisque le frontend accède au backend via le chemin `/api` dans les deux environnements (grâce au proxy Vite en dev et au reverse proxy nginx en prod), le code React n'a pas besoin de changer entre les environnements. Il suffit de toujours faire des requêtes vers `/api/...`.

# <InternalPageTitle> Comment générer les artefacts ? </InternalPageTitle>

## Introduction

Un **artefact** est le résultat d'une phase de build : c'est le fichier ou l'ensemble de fichiers prêts à être distribués ou exécutés. Dans notre application, on distingue deux types d'artefacts :

- Le **JAR** pour le backend Spring Boot.
- Le **site web statique** (`dist/`) pour le frontend React.

Ces artefacts peuvent ensuite être utilisés pour construire des **images Docker** qui encapsulent l'application et toutes ses dépendances.

## Génération du JAR pour le backend (Spring Boot)

Maven est l'outil de build utilisé pour le backend Spring Boot. Pour générer un JAR, on utilise la commande suivante à la racine du projet `api/` :

```bash
mvn clean package
```

Cette commande :
1. **Nettoie** (`clean`) le répertoire `target/` des builds précédents.
2. **Compile** le code source.
3. **Exécute les tests** unitaires.
4. **Package** l'application dans un fichier JAR dans le répertoire `target/`.

Si l'on souhaite ignorer les tests lors du packaging (par exemple lors de la construction d'une image Docker) :

```bash
mvn clean package -DskipTests
```

Le JAR généré peut ensuite être exécuté directement :

```bash
java -jar target/api-0.0.1-SNAPSHOT.jar
```

Ou avec un profil Spring Boot spécifique :

```bash
java -jar target/api-0.0.1-SNAPSHOT.jar --spring.profiles.active=prod
```

### Génération du JAR dans le Dockerfile

Dans le contexte Docker, la génération du JAR est intégrée directement dans le `Dockerfile` de l'API via un **multi-stage build** :

```Dockerfile
FROM maven:3.9.9-amazoncorretto-21 AS build

WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn clean package -Dno-build-failure

FROM eclipse-temurin:21-jre-jammy

WORKDIR /app
COPY --from=build /app/target/*.jar app.jar

EXPOSE 3000
CMD ["java", "-jar", "app.jar"]
```

La première étape (`build`) génère le JAR via Maven. La deuxième étape copie uniquement le JAR dans une image Java légère, sans avoir besoin de Maven. Cela réduit considérablement la taille de l'image finale.

### Génération du JAR dans la pipeline GitLab

Dans la pipeline GitLab CI/CD, la génération du JAR est réalisée dans le stage **`test`** (ou un stage **`build`** dédié). Le JAR est ensuite stocké comme artefact GitLab pour être utilisé par les stages suivants :

```yaml
api build:
  stage: build
  image: maven:3.9.9-amazoncorretto-21
  cache:
    key:
      files:
        - api/pom.xml
    paths:
      - $CI_PROJECT_DIR/.m2/repository
  script:
    - cd api
    - mvn clean package -DskipTests
  artifacts:
    paths:
      - api/target/*.jar
```

## Génération du site statique pour le frontend (React / Vite)

Vite est l'outil de build utilisé pour le frontend React. Pour générer le site statique, on utilise la commande suivante à la racine du projet `frontend/` :

```bash
npm run build
```

Cette commande :
1. **Compile** les fichiers TypeScript en JavaScript.
2. **Bundle** tous les fichiers en un ensemble optimisé.
3. **Minifie** le code JavaScript et CSS.
4. **Génère** les fichiers statiques dans le répertoire `dist/`.

Le répertoire `dist/` contient tous les fichiers nécessaires pour servir l'application frontend en production : un fichier `index.html`, des fichiers JavaScript et CSS minifiés, et les assets (images, polices...).

Ce répertoire peut être servi par n'importe quel serveur de fichiers statiques, comme **nginx**.

### Génération du site statique dans le Dockerfile

Comme pour le backend, la génération du site statique est intégrée dans le `Dockerfile` du frontend via un multi-stage build :

```Dockerfile
FROM node:20-alpine AS build

WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:alpine AS prod

WORKDIR /usr/share/nginx/html
COPY --from=build /app/dist .
COPY nginx.conf /etc/nginx/nginx.conf

EXPOSE 5172
CMD ["nginx", "-g", "daemon off;"]
```

La première étape (`build`) génère le site statique via Vite. La deuxième étape copie uniquement les fichiers statiques dans une image nginx très légère.

### Génération du site statique dans la pipeline GitLab

Dans la pipeline GitLab CI/CD, la génération du site statique est réalisée dans le stage **`build`**. Les fichiers générés sont stockés comme artefacts GitLab :

```yaml
frontend build:
  stage: build
  image: node:20-alpine
  cache:
    key:
      files:
        - frontend/package-lock.json
    paths:
      - frontend/node_modules
  script:
    - cd frontend
    - npm ci
    - npm run build
  artifacts:
    paths:
      - frontend/dist/
```

## Construction des images Docker

La construction des images Docker pour les différents composants de l'application (frontend, backend) est réalisée à l'aide de **Docker Compose** :

```bash
docker compose build
```

Ou pour construire et démarrer les conteneurs en une seule commande :

```bash
docker compose up --build
```

Dans la pipeline GitLab CI/CD, la construction des images Docker peut être réalisée dans un stage **`build`** dédié :

```yaml
build docker images:
  stage: build
  image: docker:latest
  services:
    - docker:20-dind
  script:
    - docker compose build
```
